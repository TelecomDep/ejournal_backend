# Неделя 8 (18–24 апреля 2026)

## Интеграция backend+frontend, расширение attendance-сценариев и документация API

### 1) Разница с прошлой неделей

Если на Неделе 7 основной фокус был на переходе в DB-only и стабилизации базового attendance flow,
то за эту неделю акцент сместился на:
- интеграцию с фронтендом
- расширение сценариев посещаемости (multi-group, привязка к ближайшей паре)
- регистрацию студента по invite-коду
- публикацию Swagger-документации и донастройку тестовых данных


### 2) Attendance: поддержка нескольких групп и журнала

Добавлены миграции:
- `migrations/20260420110000_multi_group_attendance.sql`
- `migrations/20260420132000_add_schedule_and_attendance_journal.sql`

Что сделано:
- введены таблицы `attendance_session_groups` и `attendance_session_students`
- добавлены индексы для выборок по группе/сессии/студенту
- добавлены таблицы `lesson_times`, `schedules`, `attendance`
- добавлены view `view_next_lessons` и `view_attendance_journal`
- в API и сервисе закреплен формат работы с `group_ids`

Практический эффект:
- один attendance-сеанс теперь можно корректно запускать на несколько групп
- появилась основа для журнальных/оперативных выборок по посещаемости

### 3) Совместимость с parser-данными и нормализация расписания

Добавлена миграция:
- `migrations/20260420143000_parser_write_compat.sql`

Что сделано:
- добавлены `week_type` и `day_idx_raw` в `schedules`
- реализована нормализация `day_idx` через триггер `normalize_schedule_day_idx`
- добавлены триггеры синхронизации sequence для `groups` и `teachers`
- добавлены ограничения и индексы для более стабильной записи parser-данных

Практический эффект:
- импорт/запись расписания из parser-потока стал безопаснее
- снизился риск рассинхронизации ID и ошибок с day index

### 4) Регистрация студента по invite-коду из БД

Добавлена миграция:
- `migrations/20260421100000_student_invite_registration.sql`

Что сделано:
- в `students` добавлены поля `invite_code`, `invite_code_hash`, `invite_code_used_at`
- добавлены уникальные индексы по `invite_code` и `user_id`
- реализован триггер хеширования/нормализации invite-кода
- добавлен endpoint `POST /register/by-invite`
- в сервис добавлена обработка action `register_by_invite`

Практический эффект:
- студента можно регистрировать не только по `role_hash`, но и по персональному коду приглашения из БД
- появился контролируемый сценарий «выдача кода → активация аккаунта»

Что сделано:
